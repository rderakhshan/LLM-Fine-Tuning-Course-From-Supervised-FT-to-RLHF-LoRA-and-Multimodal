Big Picture — How Pre-training Works

◆ The Pipeline — End to End

Pre-training follows a linear data-to-weights pipeline executed at enormous scale across thousands of GPUs for weeks or months:

Collect raw text → Clean & filter → Tokenize → Chunk to context window → Autoregressive training → Base model checkpoint
  • 1 Collect: Crawl the internet (Common Crawl), ingest Wikipedia, books, research papers, code repositories, forums. Scale: tens of terabytes to petabytes of text.
  • 2 Clean & filter: Deduplicate (MinHash), remove low-quality pages (perplexity filtering), strip HTML, filter toxic/spam content, language-identify to keep target languages.
  • 3 Tokenize: Convert text to token IDs using a learned tokenizer (BPE, SentencePiece). Vocabulary size is typically 32K–128K tokens.
  • 4 Chunk: Split the tokenized corpus into contiguous blocks matching the model's context window (e.g., 2048, 4096, or 8192 tokens). Documents are concatenated with a special separator token.
  • 5 Train: Autoregressive language modeling — for each position, predict the next token given all previous tokens. Loss is cross-entropy over the vocabulary.
  • 6 Checkpoint: Save model weights periodically. The final checkpoint is the base model.
Causal LM Autoregressive Cross-entropy loss BPE tokenizer Deduplication

■ Data Types & Sources

The pre-training corpus is a mixture of diverse text sources, carefully balanced to produce a general-purpose model:

  • W Web pages — Common Crawl, filtered and deduplicated. The largest source (60–80% of tokens).
  • B Books — Project Gutenberg, licensed books. Long-form narrative and deep reasoning.
  • A Academic papers — arXiv, PubMed Central. Scientific rigor.
  • C Code — GitHub repositories. Logical structure and syntax.
  • E Encyclopedic — Wikipedia in multiple languages. Factual knowledge.
  • F Forums — Reddit, Stack Overflow. Conversational patterns.
  • M Multilingual — Non-English text, 30–100+ languages.

Data ratios matter — too much web text makes the model chatty; too much code makes it overly literal. Each team has proprietary mixing ratios.

Common Crawl Wikipedia GitHub arXiv Reddit Multilingual

■ What the Process Actually Does

At the mathematical level, pre-training solves one objective: maximize the log-probability of each token given its preceding context across the entire corpus.

  • → Loss function: L = -∑ log P(tokeni | token1, ..., tokeni-1)
  • → Transformer processes the entire sequence in parallel via masked self-attention
  • → Training is distributed across hundreds/thousands of accelerators (data + tensor + pipeline parallelism)
  • → Optimizer: AdamW with cosine LR schedule, warmup, weight decay. Batch sizes in millions of tokens.
  • → Multiple epochs (1–3 passes over the corpus), loss steadily decreases
  • → Emergent abilities appear past >6B parameters: in-context learning, chain-of-thought, world modeling
Autoregressive Distributed training AdamW Cosine schedule Emergent abilities

■ What the Model Learns

The model's weights encode a compressed representation of the training distribution. It implicitly learns:

  • K World knowledge — facts, entities, dates, events, scientific concepts.
  • L Language competence — morphology, syntax, semantics, discourse, style across dozens of languages.
  • R Reasoning patterns — logical chaining, analogies, step-by-step inference learned implicitly from data.
  • C Cultural & social norms — conventions and biases present in the training text.
  • T Transferable representations — hidden states serve as feature extractors adaptable with minimal fine-tuning data.

Critically, the base model cannot: follow instructions, hold conversations, refuse harmful requests, cite sources, or produce structured outputs. These require SFT and alignment.

Key Takeaways — Unsupervised Pre-training

Why It Matters CORE INSIGHT

Unsupervised pre-training is the foundational stage that produces the base model through self-supervised learning — the model labels its own data by predicting the next token. This stage is where the LLM earns its name: doing language modeling at large scale (trillions of tokens) is what makes it a Large Language Model.

Unsupervised Mechanism

  • No human labels needed — the next word in the sequence is the label
  • Input: "Metformin improves insulin" → Target: "sensitivity"
  • The model learns purely from raw text without any annotated dataset
  • Same mechanism is reused in non-instructional fine-tuning (domain adaptation)
Unsupervised Next-token prediction No annotation

Scale & Infrastructure Barrier

  • Requires internet-scale data: Common Crawl, Wikipedia, books, code, research papers
  • Demands massive GPU clusters — hundreds of thousands of GPU-hours
  • Only large organizations can afford it: Meta, OpenAI, Google, DeepSeek, Mistral
  • This is why practitioners never train from scratch — they take existing base models and fine-tune

What the Model Actually Learns

  • General knowledge about the world (facts, concepts, relationships)
  • Language mechanics: grammar, syntax, sentence structure
  • Reasoning patterns implicit in the training text
  • But it does NOT learn to follow instructions, hold conversations, or produce structured answers — that requires the next stages

Context Window & Chunking Strategy

  • Each model has a max context window (e.g., GPT-3: 2K tokens, GPT-4: 128K tokens)
  • Pre-training data is split into chunks that fit within this window
  • Chunking can be done by: paragraph boundaries, semantic meaning, or fixed token length
  • Proper chunking strategy is critical for effective domain adaptation fine-tuning
Context window Chunk size Token limits

Practical Implications for Fine-tuning

  • You cannot do this stage yourself — use pre-trained base models from Hugging Face
  • Base model naming: llama-3.1-8B (base) vs llama-3.1-8B-instruct (fine-tuned)
  • Domain adaptation does re-use the same next-token prediction mechanism
  • Understanding pre-training helps debug fine-tuning failures (e.g., base model gaps)

Stage 2 — Supervised Fine-tuning (SFT)

Two Distinct Goals
Domain-adapt or teach instruction-following — two data strategies, one parameter-efficient toolkit.
Non-instructional
Adapt model to a specific domain's language, tone, and terminology from plain text (PDFs, documents).
help model learn the language, tone, and terminology specific to your domain
Instructional
Teach the model to follow instructions, answer questions, and produce structured human-like responses.
to train the model to understand instruction and generate well-structured and relevant answer

⚖ Parameter Level HOW

Full fine-tuning updates all weights — requires multi-GPU. Partial fine-tuning trains a subset. Modern LLMs use PEFT — train only a small set of extra parameters while the base model stays frozen.

Goal: Efficient adaptation Pro: Universal standard Con: Rank selection matters

Mechanism: Injects trainable low-rank decomposition matrices A and B into attention projection layers (Q, K, V, O). The original weights stay frozen. Forward pass: h = W0x + (B·A)x. Rank r controls expressiveness vs parameter count.

Pros: De facto standard PEFT method. Works across all model sizes and tasks. Adapters are portable (few MB). Mergeable into base weights at inference (zero overhead).

Cons: Sensitivity to rank r and target module selection. Too low rank limits capacity; too high defeats memory saving. Requires tuning alpha, dropout, and which layers to target.

Goal: Memory-efficient fine-tuning Pro: 65B on 1 GPU Con: Quantization noise

Mechanism: Applies standard LoRA on top of a 4-bit or 8-bit quantized base model. The base weights are stored in NF4 (normalized float 4) precision and dequantized on-the-fly during forward/backward. LoRA adapters remain in full precision (FP16/BF16).

Pros: Fine-tune a 65B model on a single 48GB GPU. 4-bit quantization cuts memory by ~4x vs FP16. LoRA adapters stay high-precision, so quality loss is minimal (within 1-2% of full LoRA).

Cons: Quantization introduces slight numerical noise. Training is marginally slower due to on-the-fly dequantization. Double quantization (DQ) adds complexity. Not all hardware supports NF4 efficiently.

Goal: Better fidelity than LoRA Pro: Learns direction + magnitude Con: Slightly more params

Mechanism: Weight-Decomposed Low-Rank Adaptation. Decomposes the weight update into magnitude (a learnable scalar vector m) and direction (a LoRA-style low-rank update). Formally: W' = m · (W0 + ΔW) / ||W0 + ΔW||. The magnitude vector m is learned independently.

Pros: Consistently outperforms LoRA on reasoning and knowledge-intensive tasks. Learns both the direction and scale of weight updates, mimicking full fine-tuning more closely. Minimal parameter overhead (only d extra scalars).

Cons: Slightly more compute per step due to normalization. Still relatively new — fewer community recipes and ablation studies compared to LoRA.

Goal: Insert & tune bottleneck layers Pro: Parallelizable Con: Inference latency

Mechanism: Small trainable bottleneck modules are inserted into each transformer block, typically after the attention or FFN sub-layer. Each adapter consists of a down-projection (d → r) and an up-projection (r → d) with a non-linearity in between. The original weights are frozen.

Pros: Pioneered the PEFT concept. Can be placed in parallel or sequentially. Bottleneck structure forces information compression. Some adapter variants (e.g., Parallel Adapter) add no sequential depth.

Cons: Adds inference latency (extra forward pass through adapter layers). Less parameter-efficient than LoRA for the same capacity. Largely superseded by LoRA in practice.

Goal: Minimal-parameter tuning Pro: Tiny footprint (0.1%) Con: Limited capacity

Mechanism: Only the bias terms (about 0.1% of total parameters) of the model are trainable. All weight matrices are frozen. The bias vector in each Linear and LayerNorm layer is updated via standard gradient descent.

Pros: Extremely lightweight — memory and compute overhead is negligible. Surprisingly effective for small domain shifts (e.g., adapting to a specific writing style or genre). Fastest PEFT method to train.

Cons: Limited capacity for complex task adaptation. Cannot learn new knowledge or substantially alter model behavior. Only works well when the base model already has relevant capabilities.

Goal: Rescale activations Pro: Extremely few params Con: Limited expressiveness

Mechanism: Infused Adapter by Inhibiting and Amplifying Inner Activations. Learns element-wise scaling vectors lk, lv, lf that rescale the keys, values, and FFN activations respectively. Each vector has the same dimension as the corresponding activation. Formally: K' = lk ⊙ K, V' = lv ⊙ V.

Pros: Parameter count comparable to BitFit (often even fewer). Conceptually simple — just learns which activations to amplify or suppress. Works well for classification-style tasks.

Cons: Does not add new parameters or computation paths — only re-weights existing activations. Limited expressiveness for generative tasks requiring new knowledge or behaviors. Rarely used for LLM chat fine-tuning.

Goal: Condition via virtual tokens Pro: Per-layer conditioning Con: Shrinks context window

Mechanism: Prepends a set of learnable virtual token embeddings to the keys and values of each attention layer. Unlike prompt tuning (input layer only), prefix tuning modifies attention at every layer. The prefix tokens attend to all real tokens and vice versa.

Pros: More expressive than prompt tuning because every layer is conditioned. Virtual tokens can encode task-specific behavior. Works well for generation tasks (summarization, translation).

Cons: Each prefix token consumes space in the effective context window (prefix length + real tokens must fit within max sequence length). Harder to optimize — prefix parameters can be unstable early in training. Slightly more parameters than prompt tuning.

Goal: Soft prompt conditioning Pro: Simplest PEFT Con: Input layer only

Mechanism: Prepends a sequence of learnable soft prompt embeddings to the input embedding layer only. Unlike prefix tuning (per-layer), the soft prompt is concatenated with the input tokens at the very first layer, then processed normally through all transformer layers.

Pros: Simplest PEFT method conceptually. Very few parameters (soft prompt length × embedding dim). Soft prompts are portable and can be shared/combined. No inference overhead if pre-computed.

Cons: Only the input layer is conditioned — later layers are not directly modified. Less expressive than prefix tuning for complex tasks. Soft prompt length consumes context window space. Performance degrades if prompt length is too short.

LoRA is the default choice QLoRA for memory-constrained Single GPU feasible

☷ Data Level WHAT

Non-instructional (domain adaptation): Plain text from PDFs, documents, books. Trained via the same next-token prediction mechanism as pre-training — the model learns domain-specific vocabulary, tone, and terminology.

  • → Example: Pharma PDF about Metformin → model becomes a pharma domain expert
  • → Data needs: chunking by paragraph/token limit, cleaning, formatting as text column
  • → Output: produces continuous text in-domain but does NOT learn to follow instructions

Instructional (chat/conversational): Question-answer pairs — the model learns to map a user instruction to a structured, helpful response.

  • → Formats: Alpaca (instruction + input + response), context/response, user/assistant, system/user/assistant
  • → Datasets: Alpaca, Open-Orca, Mental Health Counseling
  • → Data creation: manual annotation, expert labeling, or LLM synthetic generation (feed plain text to GPT and ask it to generate Q&A pairs)
  • → This is the stage that gives SFT its name — because the data has a human-curated label (the answer)
Domain adaptation Chatbot construction Synthetic data via LLM Alpaca format
Chronological Order — The 4-Step Pipeline
1   Pre-training → 2   Domain FT → 3   Instruction FT → 4   Alignment (DPO)

Why step 2 before step 3? If you skip domain FT, the model doesn't understand your domain's vocabulary — it would be answering questions about terms it can't grasp. The instructor calls this out explicitly: "this is not a good practice — directly to teach your model on input and output." The recommended strategy: non-instructional FT first to embed domain terminology into the weights, then instructional FT to teach conversation on top of that foundation. Vocabulary immersion before conversation class.

Stage 3 — Preference Alignment

Goal
Align model outputs with human expectations — make responses safe, helpful, polite, and honest rather than just factually correct.
to make the model polite, safe, helpful, and aligned with the human value — improve the safety and ethics of the model
Goal: Human-aligned RL Pro: High quality signal Con: Complex, 4 models

Mechanism: Three-step pipeline: (1) Collect human rankings of multiple model outputs for the same prompt. (2) Train a reward model that predicts which output humans prefer. (3) Use PPO (Proximal Policy Optimization) to update the LLM to maximize the reward model's score while staying close to the original SFT model via a KL penalty.

Pros: Human feedback provides high-quality preference signal. The reward model can generalize to unseen prompts. Pioneered by OpenAI's InstructGPT and proven at scale.

Cons: Complex to train — requires 4 models (policy, reference, reward, value) in memory simultaneously. PPO is notoriously unstable. Human annotation is expensive and hard to scale. The reward model can be hacked (reward hacking).

PPO algorithmReward modelHuman annotationInstructGPT
Goal: Scalable preferences Pro: AI labels millions Con: AI judge bias

Mechanism: Same RL framework as RLHF (reward model + PPO), but the preference labels are generated by an AI judge (e.g., a larger LLM like Claude or GPT-4) instead of human annotators. The AI is given a rubric (e.g., "constitutional principles") and asked to rank which of two model outputs is better.

Pros: Much more scalable — AI can label millions of preferences at a fraction of human cost. Introduced by Anthropic in their "Constitutional AI" paper. Enables iterative alignment during training without hiring annotators.

Cons: May inherit the AI judge's own biases and blind spots. If the judge is a weaker model, the signal is noisy. Still uses PPO with all its complexity (reward model, 4 models in memory, instability).

ScalableAI-as-judgeConstitutional AIAnthropic
Goal: Simplified alignment Pro: No reward model Con: Needs quality pairs

Mechanism: The current state-of-the-art. A supervised approach — no reward model, no reinforcement learning loop, just a clever loss function. The loss maximizes the gap between the model's likelihood of the chosen vs rejected response, relative to the reference (SFT) model:

&calL; = − 𝔼(x,y+,y−) log σ( β log πθ(y+|x)/πref(y+|x) − β log πθ(y−|x)/πref(y−|x) )

β (beta) controls how strongly the model must favor chosen over rejected. Higher β = stronger separation.

Pros: Much simpler to train — only 2 models in memory (policy + reference), no reward model, no PPO instability. Directly optimizes the preference objective. Converges faster and more stably than RLHF. Industry standard since 2024.

Cons: Requires high-quality preference pairs — noisy labels degrade performance. Does not explore new behaviors (the way RL does). The reference model must be frozen throughout training.

M Meta — Llama

Pre-training on trillions of tokens (Common Crawl, books, code) → SFT (instruction fine-tuning on QA data) → RLHF / DPO for alignment. Meta releases both base and chat/instruct variants openly on Hugging Face.

Base → SFT → RLHF → Llama-3-Chat

G OpenAI — GPT / InstructGPT

GPT-3 (2020) was pre-trained only → InstructGPT (2021) added SFT (on human-written prompts + responses) then RLHF via PPO → GPT-3.5 / GPT-4 heavily aligned with RLHF plus safety filters and constitutional guardrails. Closed source — API-only access, no base model release.

M Mistral AI

Mistral 7B (2023) base model → Mixtral 8x7B mixture-of-experts base → instruction versions via SFT → some variants further aligned with RLHF / DPO. Open weight release — both base and instruct models available on Hugging Face.

7B Base → MoE Base → Instruct SFT → RLHF/DPO

D DeepSeek

DeepSeek Base (pre-trained on massive Chinese + English corpus) → DeepSeek Code (SFT on code data) → DeepSeek R1 (aligned with RLHF/human feedback for reasoning). China's major open-source push — full weights available on Hugging Face.

Base → Code SFT → R1 Alignment

G Google — Gemini

Follows the same three-stage pipeline: pre-training on Google's massive data, SFT for instruction-following, and RLHF / multi-modal alignment. Cloud-based API — no open weights. Gemini models are also aligned for safety across text, vision, and audio modalities.

Pre-training → SFT → RLHF + Safety → Gemini

Special Topics

SLM Fine-tuning

Fine-tuning Small Language Models (sub-3B parameters like TinyLlama, Phi-3, Gemma-2B) for resource-constrained or on-device deployment. Same techniques — LoRA, DPO, instruction tuning — applied at smaller scale.

Key challenge: small models have limited capacity, so data quality matters even more.

Multimodal Fine-tuning

Extending fine-tuning to vision-language models (LLaVA, Qwen-VL, InternVL). Involves aligning a vision encoder (e.g., CLIP, SigLIP) with a language model via a projection layer, then tuning on image-text instruction data.

Key difference: the loss is still next-token prediction, but the input now includes visual tokens.

Embedding Fine-tuning

Tuning embedding models for improved retrieval quality in RAG pipelines. Uses contrastive learning on paired (query, relevant document) vs (query, irrelevant document) data to pull relevant embeddings closer and push irrelevant ones apart.

Different from LLM fine-tuning — no next-token prediction, only representation learning.

Dataset Types

☶ Non-Instructional

Single plain text column. Each row is a chunk of text (paragraph, fixed token length). Used for domain adaptation and continued pre-training via next-token prediction.

  • → FineWeb — 15-trillion token web scrape (Hugging Face)
  • → Pile PubMed — biomedical research paper abstracts
  • → Custom PDFs — extract text, chunk by paragraph or token limit

☶ Instructional

Input + output pairs. Multiple column formats are supported: instruction/input/response, context/response, user/assistant, or system/user/assistant.

  • → Alpaca — 52K GPT-generated instruction-following examples
  • → Open-Orca — 4M+ examples with system prompts, questions, and responses
  • → Mental Health Counseling — context/response format for domain-specific chatbot training

☶ Preference / DPO

Prompt + chosen + rejected columns. The chosen response is the human-preferred one; the rejected is the dispreferred one. Used for DPO, RLHF, and RLAIF training.

  • → Anthropic HH-RLHF — 170K helpful/harmless dialogues with chosen/rejected pairs
  • → UltraFeedback — 64K prompts with 4 LLM responses each, rated across multiple criteria
  • → Math-Step-DPO-10K — step-level preference pairs for math reasoning
